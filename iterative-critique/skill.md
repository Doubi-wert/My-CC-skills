# Iterative Critique Skill

## 概述

本skill实现一个**批判性迭代改进循环**（Critique-Revision Loop），使用两个独立代理协同工作：

- **Generator (G)**: 基于参考文档生成内容
- **Critic (C)**: 从批判角度审查生成内容是否符合参考文档

两者通过迭代反馈循环不断改进，直到反馈减少到阈值以下。

**泛用性原则**: 本skill设计为通用框架，不针对特定项目，可适用于任何需要迭代批判的场景。

## 工作流程

### 阶段1: 初始化

1. **检查参考文档**: 判断是否有参考文档输入
   - **有参考文档**: 验证文件存在并提取关键内容
   - **无参考文档**: 基于任务目标/初始输入作为评判标准，**明确提示用户**：无参考文献输入，将以初始任务/目标为批判标准
2. **提取关键内容**: 如果参考文档过长，提取与任务相关的章节/段落
3. **理解任务**: 明确G需要生成什么（如教学文档、代码实现、报告等）
4. **建立评判标准**:
   - 有文档时：由文档内容建立评判标准
   - 无文档时：由任务目标/初始输入建立评判标准，C将其作为"参考文档"的替代

### 阶段2: 迭代循环

```
                    ┌─────────────────────────────┐
                    │      迭代循环 (最多N次)       │
                    │                             │
    ┌───────────┐   │   ┌─────────────────────┐   │   ┌───────────┐
    │ G生成内容  │──▶│──▶│ C批判性对比参考文档  │──▶│──▶│ G修订内容  │
    └───────────┘   │   └─────────────────────┘   │   └───────────┘
                    │             │               │
                    │             ▼               │
                    │      反馈点数量               │
                    │        < 阈值?               │
                    │             │               │
                    │             ▼               │
                    │        退出循环              │
                    └─────────────────────────────┘
```

**每次迭代包含**:
1. G基于当前版本生成修订内容（带版本号v{N}）
2. C批判性对比参考文档，给出具体修改点
3. 统计反馈点数量
4. 根据收敛标准判断是否退出（详见**收敛标准**章节）

### 阶段3: 输出

输出最终版本 + 修改记录摘要

## 无参考文档模式

当用户未提供参考文档时，系统进入无参考文档模式：
- G的生成内容需符合任务目标
- C的评判标准基于任务目标本身（而非外部文档）
- 迭代收敛后输出符合任务目标的高质量内容
- **系统会明确告知用户当前处于无参考文档模式**

**参考文档自动识别**:
系统自动识别用户提供的文件路径，识别后**必须经过用户确认**才可使用：
1. 当前工作目录下的相对路径
2. 用户明确指定的绝对路径
3. 常见的参考文档格式：.md, .txt, .pdf, .doc, .docx

## 代理配置

### Generator (G) 代理

**角色**: 内容生成者

**输入**:
- 参考文档内容（提取后的关键部分）；无文档时为任务目标
- 任务描述
- 上一轮C的反馈（如有）
- deferred_points（从之前轮次推迟的反馈点）

**输出**:
- 修订后的内容（带版本号标识，如v1, v2, v3）
- 修改说明（明确说明做了哪些改动，以及对C反馈的处理）

**行为约束**:
- 必须逐条回应C的反馈，说明接受或拒绝
- 拒绝某条反馈时需给出理由
- deferred_points必须在本轮处理，不能无限推迟
- 每个版本输出必须带版本号标识

### Critic (C) 代理

**角色**: 批判性审查者

**输入**:
- 参考文档内容（提取后的关键部分）；无文档时为任务目标本身
- G生成的内容
- 上一轮G对反馈的回应（用于检查是否正确修订）
- 已拒绝反馈列表（用于避免重复提出相同问题）

**输出**:
- 具体修改点列表（每条反馈需明确指出问题所在和位置）
- 反馈点计数
- 简短总结（内容是否符合参考，整体质量如何）

**判断原则**:
1. 基于参考文档的具体内容，不使用主观偏好
2. 无文档时，基于任务目标进行评判
3. 同一问题只反馈一次，基于(issue描述+location)的**语义相似度**判定，非精确字符串匹配
4. 每条反馈需附带修改建议
5. 反馈需具体可操作，非泛泛评价
6. **当反馈点为0时，需进行审查完整性确认**

## 收敛标准

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 最大迭代次数 | 5次 | 防止无限循环 |
| 反馈点阈值 | 3个/轮 | 单轮超过此值则继续迭代 |
| 连续低阈值轮次 | 2轮 | 连续2轮反馈点<阈值且>0则退出 |
| 单轮最大处理反馈数 | 5个 | 超出部分自动deferred到下一轮 |

**收敛判断逻辑**:
```
if 迭代次数 >= 最大迭代次数:
    退出
elif 反馈点数量 == 0:
    # 快速收敛路径：反馈点为0属于特殊边界情况
    C确认审查完整性
    if 完整:
        下一轮仍为0则退出
    else:
        要求重新审查，继续迭代
elif 反馈点数量 < 阈值 and 反馈点数量 > 0:
    # 普通低阈值路径：反馈点小于阈值但不为0
    低阈值轮次 += 1
    if 低阈值轮次 >= 2:
        退出
    else:
        继续迭代
else:
    # 反馈点数量 >= 阈值
    低阈值轮次 = 0
    继续迭代
```

**两种收敛路径的区别**:
- **快速收敛路径（反馈点=0）**: 需要C主动确认审查完整性，属于特殊边界情况
- **普通低阈值路径（0 < 反馈点 < 阈值）**: 仅需连续满足轮数条件即可退出

## 边界情况处理

### BC-1: 空反馈处理
- 如果反馈点为空但质量分数不达标，G必须生成至少1个主动优化建议
- 如果反馈点为空且质量达标，正常退出

### BC-2: 反馈过多处理（>10个）
当反馈点数量超过阈值时，按优先级选择处理，详见**反馈点优先级与分级处理**章节。

### BC-3: 重复反馈处理
- 同一反馈点被重复提出≥3次，标记为blocked，触发人工介入
- **同一问题判定**：基于(issue描述+location)的语义相似度判定

### BC-4: 质量分数下降处理
当质量分数下降超过阈值时，可配置的处理策略：

```pseudo
三种质量跟踪模式（通过 user_requirements.quality_tracking_mode 配置）:
# 'disabled': 不跟踪质量分数
# 'relaxed': 仅当整体分数下降 > threshold * 1.5 时暂停
# 'strict': 整体或任一维度分数下降 > threshold 时暂停

threshold通过 user_requirements.quality_threshold 配置（默认0.1）
```

### BC-5: G拒绝反馈的合理性判定
当G拒绝某条反馈时：
- **拒绝理由合理**: 该反馈关闭，后续不再追究
- **拒绝理由不充分**: 标记为"无效拒绝"，要求G重新处理

### BC-6: Deferred连续限制
- Deferred反馈连续deferred次数达到max_consecutive_deferred（默认3次）时，强制升级处理
- 该问题必须在本轮处理，不再允许deferred

## 严重错误定义

| 错误类型 | 触发条件 | 处理方式 |
|---------|---------|---------|
| CE-1: 格式完全错误 | 输出格式与要求完全不匹配 | 立即退出 |
| CE-2: 关键需求违背 | evaluation_criteria中标记为critical:true的需求未满足 | 立即退出 |
| CE-3: 安全漏洞 | 检测到SQL注入/XSS/命令注入等 | 立即退出 |
| CE-4: 编译/解析错误 | 代码无法编译或文档无法解析 | 立即退出 |
| CE-5: 死循环/超时 | 单次操作超过timeout阈值 | 立即退出 |

### 人工介入机制

以下情况触发人工介入：
1. 同一反馈被重复提出≥3次
2. Deferred反馈连续deferred超过2轮
3. G连续拒绝反馈≥3次且理由不充分
4. 质量分数持续下降超过阈值

**人工介入流程**:
1. 暂停迭代
2. 向用户说明当前状态和待决策事项
3. 等待用户指示
4. 根据指示继续迭代或终止

### API过载处理

当子代理调用因API限流、超时或临时故障失败时，自动触发重试：

```pseudo
配置参数:
# max_retries: 最大重试次数（默认3）
# base_delay: 初始重试延迟秒数（默认5）
# max_delay: 最大延迟上限（默认60）
# exponential_base: 指数退避基数（默认2）

重试逻辑:
attempt = 0
while attempt < max_retries:
    result = call_subagent()
    if result.success:
        return result
    if is_non_retryable_error(result.error):
        raise result.error

    delay = min(base_delay * (exponential_base ^ attempt) + random_jitter(), max_delay)
    wait(delay)
    attempt += 1

raise MaxRetriesExceeded(result.error)
```

**可重试错误类型**: `rate_limit`, `timeout`, `connection_error`, `service_unavailable`
**非重试错误类型**: `authentication_error`, `invalid_request`, `permission_denied`

### 子代理失败处理

当G或C代理连续失败超过阈值时：

```pseudo
consecutive_failures = 0
failure_threshold = 3

on_subagent_failure():
    consecutive_failures += 1
    if consecutive_failures >= failure_threshold:
        pause_current_file()
        add_to_retry_queue(current_file, reason="consecutive_failures")
        move_to_next_file()
    else:
        wait_and_retry()
```

## 反馈点优先级与分级处理

### Severity分级定义

| 等级 | 标签 | 定义 | 处理策略 |
|------|------|------|---------|
| Critical | `critical` | 关键需求违背、安全问题、格式完全错误 | 立即处理，阻塞式 |
| High | `high` | 严重影响功能的问题 | 本轮优先处理 |
| Medium | `medium` | 功能正常但有改进空间 | 本轮处理，延后也可 |
| Low | `low` | 小的优化建议 | 标记为deferred，后续轮次处理 |

### 优先级计算

三种模式（通过 user_requirements.priority_mode 配置）：
- `severity_only`: 仅按severity排序
- `scope_only`: 仅按影响范围排序
- `weighted`: 混合权重计算

### Deferred反馈跨轮次处理

```pseudo
每轮结束时:
1. 将未被处理的非Critical反馈标记为deferred
2. deferred_points记录到shared_state，包含consecutive_count
3. 下一轮开始时，G必须先处理所有deferred反馈

强制升级规则:
- consecutive_count >= max_consecutive_deferred（默认3次）时强制处理
- 连续deferred超过2轮，触发人工介入
```

## 使用示例

### 输入

用户: "基于论文'Variational tetrahedral meshing.md'，帮我生成一份MATLAB教学指南。用批判性视角检查是否符合论文，并在迭代中不断改进。"

### 处理流程

1. **初始化**: 读取论文，提取关键章节（原理、公式、算法步骤）
2. **G第1轮(v1)**: 生成初始教学指南
3. **C第1轮**: 发现5个不一致
4. **G第2轮(v2)**: 根据反馈修订，说明拒绝某条反馈的原因
5. **C第2轮**: 验证修订是否正确，剩余3个问题
6. **G第3轮(v3)**: 继续修订
7. **C第3轮**: 1个问题
8. **G第4轮(v4)**: 修订
9. **C第4轮**: 0个问题，C确认审查完整性后进入下一轮
10. **C第5轮**: 仍为0，**退出**
11. **输出**: 最终教学指南 + 修改记录 + 统计信息

## 关键原则

1. **独立性**: G和C有独立的上下文，不互相干扰
2. **具体性**: C的反馈必须具体，指出确切位置和问题
3. **可操作性**: 每条反馈需附带修改建议
4. **可追溯性**: G必须对每条反馈做出回应（接受/拒绝+理由）
5. **收敛保证**: 设置多层防护防止无限循环
6. **泛用性**: 本skill不针对特定项目，适用于任何迭代批判场景
7. **子代理强制调用**: 审查和修改功能必须且只能通过子代理调用
8. **版本追踪**: 每个版本必须带版本号，便于追踪内容演进
9. **拒绝合理性判定**: G拒绝反馈后，C需判定拒绝理由是否充分
10. **同一问题判定**: 基于语义相似度，非精确字符串匹配

## 注意事项

### 参考文档处理
- **长文档**: 提取与任务相关的关键章节，而非全量传递
- **格式验证**: 先检查文件是否为有效格式（.md, .txt, .pdf等）
- **用户确认**: 自动识别后必须经过用户确认才可使用

### G拒绝反馈的处理
G拒绝某条反馈时需满足以下条件之一：
- 参考文档中无相关依据
- 建议与任务目标不符
- 建议会导致内容逻辑矛盾

G不应因"实现困难"或"个人偏好"拒绝反馈。

### 修改记录规范
- **"谁执行，谁记录"原则**: 执行修改的代理负责记录
- **Append-only语义**: 使用追加日志而非覆盖，支持审计追溯
- **版本记录**: 每个生成版本必须记录版本号和对应迭代轮次

### 质量追踪（可选）
通过`user_requirements.quality_tracking_mode`配置：
- `disabled`: 不启用质量追踪
- `relaxed`: 仅监控整体质量下降
- `strict`: 监控整体和各个维度的质量下降

阈值通过`user_requirements.quality_threshold`配置（默认0.1）。
